# Milestone v2.0: Portal & AI Expansion

**Status:** ✅ SHIPPED 2026-04-14
**Phases:** 18, 20, 21 (shippable subset of original 13–22 scope)
**Total Plans:** 20

## Overview

v2.0 was originally scoped as "Feature Expansion" covering 8 feature areas across Phases 13–22. The milestone audit on 2026-04-14 found that 6 of 9 completed phases lacked verification and 4 cross-phase integration blockers remained open. Rather than ship a broken milestone, scope was reduced to the three phases that passed goal-backward verification: Enhanced AI, Client Portal, and Live Satellite/Traffic Maps.

Phases 13–17, 19, 22, and the gap-closure phases 23–28 were moved into v2.1. Their code is present on `main` and marked complete in phase records, but their verification, wiring, and integration obligations are carried forward.

## Milestone Summary

**Scope Reduction (2026-04-14):**

Original milestone v2.0 included Phases 13–22 (8 feature areas). Audit (`v2.0-MILESTONE-AUDIT.md`) found 12/39 requirements satisfied, 6 phases without VERIFICATION.md, and 4 critical integration blockers (INT-01, INT-02, INT-06, INT-07). Shipped scope reduced to Phases 18, 20, 21 — the three phases with complete goal-backward verification. Remaining phases reassigned to v2.1.

**Key Decisions (from phases 18, 20, 21):**

- **Phase 18:** Named MCP-only DTOs with `MCP` prefix to avoid collision with `SupabaseService` DTOs
- **Phase 18:** AI tool calls return structured drafts with `_action: "review_before_saving"` rather than inserting directly — user always approves before persistence
- **Phase 18:** iOS `formatDraftIfPresent()` scans assistant messages for JSON and renders gold-themed type-badge cards in place of raw JSON
- **Phase 18:** Input validation on `generate_rfi` and `draft_change_order` rejects empty-string inputs before LLM roundtrip
- **Phase 20:** Service-role Supabase client for public portal SSR bypasses RLS after token/slug validation
- **Phase 20:** INSERT-only RLS on `cs_portal_audit_log` enforces immutable audit trail (D-114)
- **Phase 20:** Soft-delete via `is_deleted`/`is_revoked` flags — no hard deletes of portal links (D-116)
- **Phase 20:** CSS sanitizer with 9 forbidden regex patterns + 30-property allowlist (T-20-10); SVG validation rejects 5 attack vectors (T-20-11)
- **Phase 20:** Budget masking checks both section-enabled AND `show_exact_amounts` flag (D-30, D-38); financial data masked to ranges, not redacted
- **Phase 20:** Webhook payloads contain only IDs + event type, no financial/PII data (T-20-31); 5s fire-and-forget delivery (T-20-32)
- **Phase 20:** Design tokens as single-file export (`web/src/lib/design-tokens.ts`) — source of truth for portal and app styling
- **Phase 21:** Append-only `cs_equipment_locations` with no UPDATE/DELETE RLS for tamper-proof location history (T-21-04)
- **Phase 21:** Database CHECK constraints enforce coordinate ranges at schema level, duplicated in API `validateCoordinates` (T-21-02)
- **Phase 21:** `cs_equipment_latest_positions` as DISTINCT ON database view for efficient latest-position queries
- **Phase 21:** `PortalSectionsConfig.map_overlays` as optional JSONB for backward compatibility with pre-Phase-21 portals; `DEFAULT_MAP_OVERLAYS` applied at page/API read time when absent
- **Phase 21:** Portal map API uses `/api/portal/map?token=` query param pattern (matches Phase 20 portal API auth convention)
- **Phase 21:** Straight-line `MapPolyline` hidden when computed road `MKRoute` exists for same route id (avoids double-render)
