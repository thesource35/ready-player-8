## Deferred Items Explicitly Carried Forward

Per RESEARCH §Out of scope — these are acknowledged gaps that are NOT blockers for Phase 29.1's AUTH-GATE-01/02/03 closure but should be addressed in future phases:

1. **JWT signature verification in `web/src/middleware.ts`** — defense-in-depth hardening for the web platform. Defer to performance-vs-security phase. Web middleware already calls `supabase.auth.getUser()` which validates server-side — the opportunity is to add a fast-path JWT signature check ahead of the Supabase round-trip.
2. **`kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` on `KeychainHelper`** — iOS Keychain access class today defaults to `AfterFirstUnlock` (backup exposure risk). Trivial one-line change, not urgent; defer to a dedicated Keychain-hardening quick task.
3. **`orgId = ""; canManage = true` hardcoded TODO in `web/src/app/projects/[id]/page.tsx:12-15`** — authorization, not authentication; out of scope for Phase 29.1's auth-gate work. Phase 22 follow-up explicitly owns this per RESEARCH §Out of scope.
4. **`SupabaseService.signIn()` error distinction (invalid credentials vs network)** — usability polish (the user sees the same "Sign in failed" message for both states today). Distinct error surfacing would require threading the Supabase error code through to AppError — not a safety gap, purely UX.
