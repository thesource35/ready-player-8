<objective>
Persist all volatile @State arrays in core views and add filter-triggered Supabase refetch for ProjectsView and ContractsView.

Purpose: Users lose all projects, contracts, market data, social network messages, and estimate line items every time the app restarts because these are plain @State arrays initialized to empty. Filtering in ProjectsView/ContractsView only filters the local array -- it never triggers a fresh server-side query. Both issues violate the core value of "no silent data loss."

Output: All 5 views persist their data via loadJSON/saveJSON, and filter changes in ProjectsView/ContractsView trigger fresh Supabase fetches.
</objective>

3. DATA-01 — Add filter-triggered Supabase refetch. Add `.onChange` modifiers for `filterStatus` and `searchText`:
```swift
.onChange(of: filterStatus) { _, newFilter in
    guard supabase.isConfigured else { return }
    Task {
        do {
            var query: [String: String] = [:]
            if newFilter != "All" {
                query["status"] = "eq.\(newFilter)"
            }
            let remote: [SupabaseProject] = try await supabase.fetch(SupabaseTable.projects, query: query)
            await MainActor.run { projects = remote }
        } catch {
            CrashReporter.shared.reportError("Projects filter refetch failed: \(error.localizedDescription)")
        }
    }
}
```

Note: The `supabase` reference -- check if ProjectsView already has a `let supabase = SupabaseService.shared` property. If not, add `private let supabase = SupabaseService.shared` to the struct.

Also check the existing `.task` or `.onAppear` block. If there's already a loadProjects() function, add filter params to it and reuse it in the onChange handler instead of duplicating fetch logic.
