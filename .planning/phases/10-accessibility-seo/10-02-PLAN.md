**A11Y-01: aria-labels on icon-only buttons** (per user decision: add aria-label directly, no wrapper components)

Audit each file for buttons/clickable elements that contain ONLY icons (emoji, SVG, symbol characters) with NO visible text. Add `aria-label="descriptive action"` directly to the element. Common patterns to find:

- Emoji-only buttons: `<button onClick={...}>X</button>` -> add `aria-label="Close"`
- Symbol buttons: buttons with only unicode symbols like arrows, X marks
- Icon toggle buttons in components (MobileNav hamburger, AngelicAssistant toggle)
- Send/submit buttons with only arrow icons
- Delete/remove buttons with only trash emoji or X

Do NOT add aria-label to buttons that already have visible text (e.g., "Save", "Submit", "Add Project"). Only icon-only buttons need it.

Example transforms:
- Chat send button with arrow icon -> `aria-label="Send message"`
- Close/dismiss button with X -> `aria-label="Close"`
- MobileNav hamburger -> `aria-label="Open navigation menu"` / `aria-label="Close navigation menu"`
- Filter/sort icon buttons -> `aria-label="Filter projects"` or similar
- AngelicAssistant floating button -> `aria-label="Open AI assistant"`
- Map overlay toggle buttons -> `aria-label="Toggle {overlay} overlay"`

**A11Y-02: Form label associations** (per user decision: `<label htmlFor>` with matching `id` on input)

Find all `<input>`, `<textarea>`, and `<select>` elements in these pages. For each:
1. Add a unique `id` attribute to the input if it doesn't have one (e.g., `id="login-email"`)
2. Either wrap the input in a `<label>` or add a `<label htmlFor="login-email">` element nearby
3. If the form already has visible text labels (like placeholder text or heading text), convert the visible text to a proper `<label>` element

Pages with forms needing labels (8 pages confirmed with inputs):
- login/page.tsx: email, password fields
- profile/page.tsx: name, email, phone, trade, company fields
- jobs/page.tsx: job posting form fields (title, company, contactEmail, etc.)
- contracts/page.tsx: contract form fields
- projects/page.tsx: project form fields
- ai/page.tsx: chat input
- rentals/page.tsx: quote request form
- verify/page.tsx: verification form fields

For visually hidden labels (when design shows no visible label but placeholder only), use:
```tsx
<label htmlFor="chat-input" style={{ position: "absolute", width: 1, height: 1, padding: 0, margin: -1, overflow: "hidden", clip: "rect(0,0,0,0)", whiteSpace: "nowrap", borderWidth: 0 }}>Message</label>
```

This is the standard "sr-only" pattern for visually hidden but screen-reader-accessible labels.

**A11Y-03: Status indicators text alternatives** (per user decision: `aria-label="Status: {value}"` alongside color)

Find all `<span>` or `<div>` elements that use color to convey status (green for active/on-track, red for delayed/overdue, gold for pending/warning, cyan for in-progress). These are elements where the color changes based on a status value.

For each status indicator, add `aria-label="Status: {value}"` and `role="status"` to the element.

Known locations from codebase scan (30+ indicators across these pages):

**clients/page.tsx:**
- Project status spans (On Track/Ahead/Behind) colored green/cyan/gold
- Material selection status (APPROVED/PENDING) colored green/gold
- Meeting open issues colored green/gold by count

**electrical/page.tsx:**
- Fiber project status (COMPLETE/IN PROGRESS/PLANNED) colored green/cyan/gold

**maps/page.tsx:**
- Active sites status (ACTIVE/DELAYED) colored green/red/gold

**schedule/page.tsx:**
- Milestone status (DONE/IN PROGRESS/UPCOMING) colored green/cyan/muted
- Critical path highlighting (red background for critical tasks)

**settings/page.tsx:**
- Integration status (CONNECTED/CONFIGURED/NOT CONFIGURED) colored green/muted

**field/page.tsx:**
- Equipment status, timecard status indicators

**training/page.tsx:**
- Certification status (CURRENT/EXPIRED/ENROLLED) colored by status

**analytics/page.tsx:**
- Risk scores and performance indicators colored by value

**market/page.tsx:**
- Market trend indicators colored by direction

**tasks/page.tsx:**
- Task priority indicators, completion status checkmarks

Pattern to apply:
```tsx
// Before:
<span style={{ fontSize: 8, fontWeight: 900, color: s.status === "ACTIVE" ? "var(--green)" : "var(--red)" }}>{s.status}</span>

// After:
<span style={{ fontSize: 8, fontWeight: 900, color: s.status === "ACTIVE" ? "var(--green)" : "var(--red)" }} aria-label={`Status: ${s.status}`} role="status">{s.status}</span>
```

Only add to elements where color conveys meaning. If the text content already clearly states the status (e.g., the span literally says "ACTIVE"), the aria-label is still valuable because it provides context ("Status: ACTIVE" vs just "ACTIVE" which could mean anything).
