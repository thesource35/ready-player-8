Install Upstash Redis packages and refactor the rate limiter to a dual-mode implementation (Redis primary, in-memory fallback). Wire rate limiting into Next.js middleware for all /api/* routes with per-route limits and standard headers. Refactor middleware session validation to decode JWT claims from cookies instead of calling Supabase getUser() on every request.

Output: Distributed rate limiting via middleware with JWT-based fast-path session validation.

1. Refactor `web/src/middleware.ts` to add two new capabilities:

   **A) Rate limiting for /api/* routes:**
   - Instead of `return NextResponse.next()` for API routes, call the new `rateLimit(identifier, pathname)` from `@/lib/rate-limit`
   - Use `request.headers.get("x-forwarded-for") || request.ip || "unknown"` as identifier
   - If `result.success === false`, return `NextResponse.json({ error: "Rate limit exceeded. Please wait." }, { status: 429 })` with rate limit headers
   - If success, call `NextResponse.next()` and append rate limit headers to the response:
     ```typescript
     const response = NextResponse.next();
     const headers = getRateLimitHeaders(result);
     Object.entries(headers).forEach(([k, v]) => response.headers.set(k, v));
     return response;
     ```

   **B) JWT-based session validation (replacing getUser() call):**
   - Create a helper function `decodeSupabaseJWT(cookieValue: string): { sub: string; email: string; exp: number } | null` that:
     - Splits the token on ".", takes index [1] (payload)
     - Decodes from base64url: `Buffer.from(payload, 'base64url').toString()`
     - Parses JSON
     - Checks `exp * 1000 > Date.now() + 60_000` (60-second buffer for PERF-07 pitfall avoidance)
     - Returns `{ sub, email, exp }` or null on any failure
   - In the middleware's protected route section:
     - Read the Supabase auth cookie (typically `sb-<project-ref>-auth-token` — look for cookies matching `sb-*-auth-token`)
     - Try `decodeSupabaseJWT()` on the cookie value
     - If decode succeeds (non-null, not expired), proceed without calling getUser()
     - If decode fails or token is expired, THEN create the Supabase server client and call getUser() as fallback (preserving current behavior for token refresh)
   - Keep the existing cookie management logic for the fallback path

2. Keep all existing public routes, static file checks, and demo mode checks unchanged.

3. The middleware matcher config stays the same.
