## Mechanism notes

**File-move workaround for 999.7 bare-timestamp drift** — same pattern used for migration `20260509001` earlier tonight. Two local files (`20260406_documents.sql`, `20260407_phase14_notifications.sql`) have bare-timestamp filenames that the Supabase CLI parses as version `20260406` and `20260407`, neither of which has a row in remote `schema_migrations`. The CLI bails on `db push` with "Found local migration files to be inserted before the last migration on remote database" until those files are temporarily moved out. Workaround:

1. `mv supabase/migrations/2026040{6,7}_*.sql /tmp/...parked2`
2. `supabase db push --linked` (now sees only the new migration as pending)
3. `mv /tmp/...parked2 supabase/migrations/`

The bare-timestamp files are restored after the push, so local state is unchanged. The 999.7 drift remains tracked.

**4 NOTICE messages during apply** — expected. The migration uses `drop policy if exists ... ; create policy ...` for idempotency. On a fresh table where the policies don't exist yet, the DROP statements no-op and Postgres logs a NOTICE. This is the desired behavior; the migration is now safe to re-run.

## Lessons learned

- The orphan-schema-with-recursive-RLS combo would have re-introduced a bug we'd just fixed. Always audit orphan files for stale patterns against current state before promoting.
- The `public.user_org_ids()` helper from `20260509001` is now the canonical pattern for any new RLS policy that filters by org membership. Document this in a future architectural note (or update CLAUDE.md's Architecture section).
- Bare-timestamp drift recurs on every `db push`. The file-move workaround works but is friction. Recommend renaming the 2 files (proper 999.7 fix) in the next session.
