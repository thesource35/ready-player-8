# Quick 260510-v81: Promote cs_report_schedules schema

## Objective

Promote `cs_report_schedules` from orphan `web/src/lib/reports/db-schema.sql` to a real numbered migration, push to remote, and verify end-to-end via the `/api/reports/cron` smoke test. Closes the schema-doesn't-exist half of backlog 999.5 (f).

## Context

Phase 19-08 (commit `5d7de2a`) shipped 4 consumers of `cs_report_schedules` (the cron route, schedule POST handler, health endpoint, notifications integration) but never promoted the schema from `web/src/lib/reports/db-schema.sql` (381 lines, 8 tables, no `IF NOT EXISTS` guards) to a numbered migration. All 4 consumers returned 500 "Failed to fetch schedules" until tonight.

The orphan's SELECT policy used the recursive `org_id in (select org_id from user_orgs where user_id = auth.uid())` pattern — the same bug fixed for `user_orgs` in `20260509001` earlier tonight. Naive promotion would have re-introduced the 42P17 recursion. The SELECT policy was rewritten to use the `public.user_org_ids()` SECURITY DEFINER helper from that fix.

Scope was deliberately limited to `cs_report_schedules` — the 7 other reporting tables in `db-schema.sql` remain orphan. Their consuming code paths exist but were never expected to work either, so leaving them creates no new regression.
